# 毕业设计整改任务清单

## 必须立即修改

1. 问题：模型接口密钥写在 `Backend/user_story_system/app/config.py`，仓库中还包含 `.env`。
   修改目标：项目中不出现任何真实密钥，本地和部署都通过环境变量配置。
   建议做法：删除代码中的明文密钥；`get_model_client()` 改为读取 `OPENAI_API_KEY`、`OPENAI_BASE_URL`、`MODEL_NAME`；移除仓库里的 `.env`；更换已暴露密钥；README 写清 `.env.example`。
   影响位置：代码 / 部署 / README

2. 问题：INVEST 评分逻辑与论文、PPT、前端展示不一致。
   修改目标：评分范围、阈值、维度分数、总分展示完全一致。
   建议做法：修正 `invest_score.py`，明确总分 0-100 或 0-60；同步调整 `need_improve()` 阈值；大模型评分解析失败时给出安全兜底；前端 `useChat.js` 直接使用后端 `final_score`，删除额外分数处理。
   影响位置：代码 / 论文 / PPT

3. 问题：需求文档解析页面可能无法显示后端返回的用户故事。
   修改目标：上传需求文档后，页面能正确显示每条需求对应的多条用户故事和验收标准。
   建议做法：检查 `DocumentStoryView.vue`，把 `selectedRequirement.user_story` 改为遍历 `selectedRequirement.user_stories`；对应展示每个 `story` 和 `acceptance_criteria`。
   影响位置：代码 / 部署

4. 问题：README 与实际项目目录和启动命令不一致。
   修改目标：按 README 可以从零启动本地服务和 Docker 服务。
   建议做法：将后端路径改为 `Backend/user_story_system`；前端启动命令改为 `npm run serve`；环境变量统一为 `MONGODB_URL`、`MONGODB_DB`、`OPENAI_API_KEY` 等；说明前端统一访问 `/api`。
   影响位置：README / 部署

5. 问题：PPT 缺少真实系统截图和测试结果。
   修改目标：答辩时能清楚展示“做了什么、怎么做、效果如何”。
   建议做法：增加文本需求生成、代码逆向、需求文档解析、项目源码生成、用户故事优化、历史记录、导出报告截图；第 15 页改成真实测试用例和结果。
   影响位置：PPT

6. 问题：测试支撑偏示例化，缺少正式测试记录。
   修改目标：论文和答辩材料能证明系统主流程真实可运行。
   建议做法：新增测试表，至少覆盖文本需求、代码片段、需求文档、项目源码、用户故事优化、PDF/Markdown 导出、历史记录；每条写输入、步骤、预期结果、实际结果、截图编号。
   影响位置：论文 / PPT / 文档

## 建议尽快修改

1. 问题：论文和 PPT 中部分智能体命名与代码不一致。
   修改目标：论文、PPT、代码中的角色名称和职责对应起来。
   建议做法：统一为 splitter、PM、QA、Architect、Appraiser、Document Requirement、Code Story、Project Story、Optimizer 等实际模块；删除或改写代码中没有独立实现的 Developer Agent 等表述。
   影响位置：论文 / PPT

2. 问题：论文/PPT 提到优先级、故事点、依赖关系等字段，但代码没有统一输出。
   修改目标：避免答辩时被认为夸大实现。
   建议做法：若短期不补代码，就在论文和 PPT 中改为“部分结构化字段/后续扩展”；若要保留亮点，就补齐后端返回、前端展示和导出内容。
   影响位置：代码 / 论文 / PPT

3. 问题：论文测试章节证据不足，当前材料不足以证明质量提升。
   修改目标：用真实数据支撑“优化有效”。
   建议做法：选 2-3 组固定输入，保存优化前用户故事、优化后用户故事、INVEST 分数变化、验收标准变化，并放入论文测试章节和 PPT。
   影响位置：论文 / PPT

4. 问题：部署平台虽然在线，但未提供完整演示脚本。
   修改目标：答辩现场能稳定演示核心流程。
   建议做法：准备固定文本需求、固定代码片段、固定需求文档和固定源码压缩包；提前验证每一步能生成结果；保留历史记录作为备用展示。
   影响位置：部署 / PPT / 文档

5. 问题：导出能力说明不统一。
   修改目标：README、前端按钮、后端接口实际支持格式一致。
   建议做法：逐项核对文本、代码、文档、项目源码、优化结果的导出格式；README 不要写未实现的 Word/PDF 导出，或补齐对应接口。
   影响位置：代码 / README / PPT

6. 问题：仓库归档不够整洁。
   修改目标：提交材料干净、可归档。
   建议做法：清理 `__pycache__`、`.idea`、本地环境文件；补充 `.gitignore`；保留必要测试数据和输出样例即可。
   影响位置：代码 / 文档

## 可选优化

1. 问题：缺少自动化测试。
   修改目标：降低答辩前临时出错风险。
   建议做法：后端增加接口级测试；前端至少增加关键页面构建检查；准备一个最小 smoke test 脚本验证 `/api/openapi.json`、`/api/agents/config`、导出接口。
   影响位置：代码 / 部署

2. 问题：源码解析对 JS/Vue/Java 的结构识别偏规则化。
   修改目标：提高项目源码生成用户故事的可信度。
   建议做法：论文中如实说明解析策略；代码层面可逐步引入更稳定的 AST/解析器，减少正则识别误差。
   影响位置：代码 / 论文

3. 问题：前端部分图表和分数说明不够清楚。
   修改目标：展示时让老师能快速理解评分变化。
   建议做法：雷达图和折线图标注评分范围、轮次、子需求编号；展示后端返回的评分理由。
   影响位置：代码 / PPT

4. 问题：用户手册和部署手册不独立。
   修改目标：归档材料更完整。
   建议做法：从 README 拆出 `deployment.md`、`user_manual.md`、`test_report.md`，分别说明部署、操作流程和测试结果。
   影响位置：文档

5. 问题：GitHub 仓库缺少项目简介和发布说明。
   修改目标：在线材料更像正式毕业设计项目。
   建议做法：补充仓库 About、README 顶部截图、版本说明、演示地址、主要功能列表和运行状态说明。
   影响位置：README / GitHub
